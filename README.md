# Виртуальная экскурсия по институту

Интерактивная 360° экскурсия по зданию института на чистом **TypeScript + Three.js + Vite**.
Без React/Vue/Svelte и без сторонних UI-библиотек — чтобы движок был полностью прозрачным.

Первый этап показывает одну equirectangular-панораму с вращением мышью/касанием,
зумом, плавной камерой, адаптивностью, полноэкранным режимом и поддержкой мобильных.
Архитектура с самого начала рассчитана на сотни панорам, переходы, миникарту,
поиск кабинетов, точки интереса, аудиогид, VR, локализацию и загрузку данных с сервера.

## Запуск

```bash
npm install
npm run dev      # http://localhost:5173
```

Прочие команды:

```bash
npm run build      # типизация + production-сборка в dist/
npm run preview    # предпросмотр production-сборки
npm run typecheck  # строгая проверка типов
npm run lint       # ESLint
```

## Структура проекта

```
public/
  assets/panoramas/main.jpg     360° исходник (статика, отдаётся как /assets/...)
src/
  app/        App — корень композиции и политика навигации
  core/        типы домена, EventBus, Disposable, контракт событий
  engine/      Renderer (WebGL + DPR + context-loss), RenderLoop (rAF)
  scene/       SceneManager — владелец графа сцены
  camera/      CameraController — yaw/pitch/fov + плавность и лимиты
  controls/    PointerControls (drag + pinch), WheelControls, ControlsManager
  panorama/    Panorama (сфера+очистка), PanoramaLoader (с отменой), Repository
  hotspots/    HotspotLayer + Hotspot — кликабельные точки переходов
  viewer/      Viewer — фасад отображения (init/loadPanorama/resize/dispose)
  ui/          UIManager + компоненты (топбар, fullscreen, оверлей загрузки, миникарта)
  config/      viewer.ts (параметры рендера), panoramas.ts (граф тура), floorplan.ts (карта этажей)
  styles/      main.css — тёмная минималистичная тема
  utils/       math.ts, fullscreen.ts
```

### Принципы архитектуры

- **Одна ответственность на класс.** Никакого God Object: загрузка текстур,
  геометрия сферы, инпут, камера, рендер-цикл и UI — отдельные модули.
- **Связь через события, а не ссылки.** UI слушает `Viewer.events`
  (типизированный `EventBus`); `Viewer` ничего не знает об UI. Граф зависимостей
  ацикличен.
- **DI в корне.** Все зависимости связываются в `App` — единственном месте, где
  модули «встречаются».
- **Данные отделены от кода.** Граф тура описан в `config/panoramas.ts` и читается
  только через `PanoramaRepository`, поэтому замена статики на запрос к серверу/CMS
  затрагивает ровно один файл.

## Как работает Viewer

`Viewer` — это **только отображение**, без бизнес-логики. Он композирует движок и
предоставляет жизненный цикл, которым управляет приложение:

```
init()           → измерить контейнер, запустить ResizeObserver и рендер-цикл
loadPanorama(node) → загрузить текстуру и атомарно заменить сферу
resize()         → обновить размер рендера и aspect камеры
dispose()        → отменить загрузку, остановить цикл, освободить все ресурсы
```

`Viewer` не решает, *какую* панораму показывать, и не знает, что делает точка
перехода — он лишь рендерит то, что ему сказали, и сообщает о произошедшем через
`events`. Политика навигации живёт в `App` (обработчик `link:activated`).

## Как устроена сцена Three.js

- **Сцена** (`SceneManager`) содержит один объект — сферу-панораму. Это
  единственная точка `add`/`remove`, к которой позже подключатся слои точек
  интереса, VR-контроллеры и т.п.
- **Камера** (`PerspectiveCamera`) стоит в центре сферы. Ввод не двигает матрицу
  напрямую: контролы пишут *целевые* `yaw/pitch/fov`, а `CameraController.update()`
  каждый кадр сглаживает *текущие* значения к целевым через кадрово-независимый
  экспоненциальный демпинг (`MathUtils.damp`). Отсюда «вес» и плавность движения.
- **Сфера** (`Panorama`) — `SphereGeometry`, вывернутая наизнанку через
  `geometry.scale(-1, 1, 1)`, с `MeshBasicMaterial`, на который натянута текстура.
  Мы смотрим на неё изнутри.
- **Рендер-цикл** (`RenderLoop`) — это голый `requestAnimationFrame`, который
  считает ограниченную `delta` и вызывает колбэк. Он ничего не знает о приложении.

## Как происходит загрузка текстур

`PanoramaLoader` не использует `THREE.TextureLoader`, потому что тот **нельзя
отменить**. Вместо этого:

1. `fetch(url, { signal })` — сетевой запрос, управляемый `AbortSignal`.
2. `Blob → ObjectURL → Image.decode()` — декодирование с поддержкой отмены.
3. Из `Image` создаётся `THREE.Texture` (`SRGBColorSpace`, максимальная
   анизотропия GPU), которую Three лениво загрузит на видеокарту при следующем кадре.

При смене панорамы или `dispose()` незавершённая загрузка `abort()`-ится, а старая
сфера освобождается **только после** готовности новой текстуры — экран не мигает.

### Управление памятью GPU

Каждая `Panorama` владеет ровно тремя GPU-ресурсами и освобождает все в `dispose()`:

```ts
geometry.dispose();
material.dispose();
texture.dispose();
```

Пропуск любого из них = утечка VRAM на каждом переходе. Это критично для длинной
мобильной экскурсии.

## Как добавить новую панораму

1. Положите изображение в `public/assets/panoramas/`, например `lobby.jpg`.
2. Добавьте узел в `src/config/panoramas.ts`:

```ts
{
  id: 'lobby',
  title: 'Холл первого этажа',
  image: '/assets/panoramas/lobby.jpg',
  initialYaw: 90,    // куда смотрит камера при входе, в градусах
  initialPitch: 0,
  links: [],
}
```

Уникальный `id` — единственное требование. Как её показать и как связать с
другими — ниже.

## Как связывать сцены и переходить между ними

Переход между панорамами состоит из **двух независимых половин**:

1. **Данные** — связь описывается в `config/panoramas.ts` как ребро графа
   (`PanoramaLink`). Это «куда можно пройти и где находится дверь».
2. **Триггер** — то, что фактически вызывает смену панорамы во время работы.
   Триггер вызывает `App.navigateTo(id)`, а тот — `Viewer.loadPanorama(node)`.

Есть два триггера, и оба уже работают:

- **Клик по точке-хотспоту** в кадре — основной способ для пользователя. За него
  отвечает слой `hotspots/`: он рисует на каждой сцене её `links` как маркеры и по
  клику запускает переход.
- **Программный** `navigateTo(id)` — для тестов, диплинков, будущей миникарты/поиска.

Достаточно правильно описать связи в данных — клик заработает сам. Ниже — как.

### Шаг 1. Модель связи (`PanoramaLink`)

```ts
interface PanoramaLink {
  targetId: string;  // id сцены, КУДА ведёт переход
  yaw: number;       // горизонтальный угол точки на сцене, в градусах
  pitch: number;     // вертикальный угол точки, в градусах
  label?: string;    // подпись/тултип (необязательно)
}
```

Важно понимать три вещи:

- **Связь направленная.** `links` у сцены `201` с `targetId: '202'` создаёт переход
  **только** `201 → 202`. Чтобы вернуться, у сцены `202` нужен свой обратный link с
  `targetId: '201'`. Двусторонняя дверь = два ребра.
- **`yaw`/`pitch` — это место точки на ТЕКУЩЕЙ сцене**, а не на целевой. Это
  направление, в котором на панораме `201` видна дверь, ведущая в `202`.
- **Углы в градусах.** `yaw` — поворот по горизонтали относительно стартового
  направления сцены (`initialYaw = 0`): `0` — прямо вперёд, положительные значения
  поворачивают в одну сторону, отрицательные — в другую (по 360° кругу). `pitch` —
  наклон: `0` — горизонт, `+` — вверх (к потолку), `−` — вниз (к полу/двери).

### Шаг 2. Двусторонняя связь — полный пример

Ваши сцены `201` и `202`. Сейчас связь односторонняя (есть только `201 → 202`).
Добавим обратное ребро и поправим подписи — получится, что между аудиториями можно
ходить туда-обратно:

```ts
export const panoramas: PanoramaNode[] = [
  {
    id: '201',
    title: 'Аудитория 201',
    image: '/assets/panoramas/201.jpg',
    initialYaw: 0,
    initialPitch: 0,
    links: [
      // На панораме 201 дверь в 202 видна правее и чуть вниз:
      { targetId: '202', yaw: 30, pitch: -5, label: 'В аудиторию 202' },
    ],
  },
  {
    id: '202',
    title: 'Аудитория 202',
    image: '/assets/panoramas/202.jpg',
    initialYaw: 0,
    initialPitch: 0,
    links: [
      // Обратный переход. Угол СВОЙ — это где дверь в 201 видна уже на панораме 202:
      { targetId: '201', yaw: -150, pitch: -5, label: 'В аудиторию 201' },
    ],
  },
];
```

> Подсказка про обратный угол: если на сцене `201` дверь в `202` находится под
> `yaw: 30`, то на сцене `202` дверь обратно почти всегда «за спиной» — примерно
> `30 ± 180`, то есть около `-150` или `210`. Точное значение зависит от того, как
> сняты фотографии, и подбирается за пару итераций (см. шаг 4).

### Шаг 3. Стартовый ракурс (`initialYaw` / `initialPitch`)

Это не про связи, но напрямую влияет на переходы: после загрузки сцены камера
мгновенно встаёт в `initialYaw/initialPitch` целевой панорамы. Выставляйте их так,
чтобы при входе пользователь сразу смотрел «вперёд по ходу движения», а не в стену.

### Шаг 4. Как подобрать `yaw`/`pitch` точки

Точных координат «из коробки» нет — их подбирают эмпирически, благо Vite
перезагружает изменения `config/panoramas.ts` мгновенно (HMR):

1. Запустите `npm run dev`, откройте нужную сцену.
2. Мышкой наведите центр экрана ровно на дверной проём, ведущий в соседнюю сцену.
3. Прикиньте углы: насколько вы повернулись вбок (`yaw`) и вверх/вниз (`pitch`)
   относительно стартового вида. Впишите в link.
4. Сохраните файл — страница обновится. Если точка «не там»: вправо/влево —
   меняйте `yaw` (или его знак), выше/ниже — `pitch`. Повторяйте, пока точка не
   ляжет на проём.

### Шаг 5. Проверка

Запустите `npm run dev` — на сцене `201` появится пульсирующий маркер. Клик/тап по
нему плавно перенесёт в `202`, где будет свой маркер обратно в `201`. Перетаскивание
для осмотра переходом не считается (есть порог `tapThreshold`).

Программный путь для тестов/диплинков — `navigateTo(id)`. Чтобы дёрнуть его из
консоли браузера, временно прокиньте приложение в `window` (только для отладки):

```ts
// в конце src/main.ts:
(window as unknown as { app: typeof app }).app = app;
```

```js
app.navigateTo('202');  // загрузит и покажет аудиторию 202
```

### Как это устроено внутри

Связи проходят сквозь слои без спецслучаев:

1. `Viewer.loadPanorama(node)` после смены сферы вызывает `HotspotLayer.setLinks(node.links)`;
2. `HotspotLayer` рисует на каждый link спрайт-маркер по его `yaw/pitch`
   (та же сферическая база, что у камеры — поэтому маркер точно там, куда смотришь);
3. клик/тап по маркеру (с отсевом перетаскиваний через raycast) публикует
   `link:activated → { targetId }`;
4. `App` (`src/app/App.ts`) ловит событие и вызывает `navigateTo(targetId)` →
   `loadPanorama` → круг замыкается.

`Viewer` остаётся чисто отображающим, а решение «куда идти» живёт в `App` — добавить
миникарту/поиск/диплинки можно через тот же `navigateTo`, ничего не переписывая.

### Как поменять картинку маркера

Внешность маркера — это одна текстура, общая на все переходы. Достаточно поменять
путь `hotspot.icon` в `src/config/viewer.ts` (по умолчанию `/assets/ui/arrow.png`,
файл лежит в `public/assets/ui/`). Позиционирование, пульсация и клик от картинки не
зависят и остаются как есть. Соотношение сторон подхватывается автоматически из
самого файла (`HotspotLayer` читает натуральные размеры текстуры после загрузки),
поэтому не-квадратную картинку не искажает: `hotspot.size` задаёт высоту, ширина
следует за пропорцией изображения. Для процедурного маркера или видео замените
тело `src/hotspots/hotspotTexture.ts` (`CanvasTexture` / `VideoTexture`).

## Миникарта и калибровка направления

В правом нижнем углу — сворачиваемая миникарта этажа: план комнат, точки-сцены,
подсветка текущей точки и **конус направления взгляда**, который крутится вместе с
камерой. Клик по точке = переход в неё (тем же событием `link:activated`, что и
3D-маркер, — навигацию по-прежнему решает только `App`). Компонент —
`src/ui/components/Minimap.ts`, он полностью управляется через шину событий
`Viewer` и читает `Viewer.cameraYaw` каждый кадр.

### Данные карты

Вся карта живёт в **одном файле** `src/config/floorplan.ts` — правьте его, всё
подхватится по HMR:

- `rooms` — прямоугольники (комнаты, коридор) с подписями; это только фон плана;
- `points` — точки: `id` **должен совпадать** с `id` сцены в `panoramas.ts`
  (именно это связывает сцену с местом на карте), плюс координаты `x/y` в системе
  `viewBox` этажа.

Схема намеренно **компактная и не масштабная** — задача карты дать понимание «где я»,
а не точный чертёж. Сцены без записи в `floorplan.ts` (например Сибкодинг, другой
этаж) просто прячут панель — новый этаж добавляется отдельным объектом в `floors`.

### Калибровка конуса (`northOffset`)

У каждой панорамы свой произвольный «север» (yaw=0 смотрит в разные стороны), поэтому
направление конуса калибруется на сцену полем `northOffset` у точки (в градусах,
по часовой, 0 = вверх карты). Как подобрать за минуту:

1. Зайдите на сцену, найдите на панораме заметный ориентир (дверь, окно), который
   узнаёте на карте.
2. Повернитесь к нему камерой и посмотрите, куда смотрит конус на карте.
3. Прибавьте к `northOffset` угол рассогласования (конус должен смотреть туда же).
   Повторяйте, пока не совпадёт. Значение сохраняется в `floorplan.ts`.

Если при вращении камеры конус крутится **в обратную сторону** — переключите
`minimap.yawSign` в `src/config/viewer.ts` на `-1` (знак общий на весь этаж, а
точное направление у каждой сцены задаёт `northOffset`). Размер конуса и точек —
там же, в блоке `minimap`.

> Точка входа по умолчанию (`entryPanoramaId`) сейчас в Сибкодинге — на нём карты
> ещё нет, поэтому панель появится, как только вы перейдёте на этаж 201–220.

## Производительность и надёжность

- **devicePixelRatio** ограничен `maxPixelRatio` (по умолчанию 2) — Retina/4K не
  множат число фрагментов впустую.
- **Resize** — через `ResizeObserver` на контейнере, а не глобальный `window.resize`.
- **Потеря WebGL-контекста** перехватывается (`preventDefault`), цикл
  останавливается и корректно перезапускается с пере-загрузкой текстуры при восстановлении.
- **Отмена загрузок** через `AbortController` исключает гонки при быстрых переходах.
- **Никаких утечек**: каждый `Disposable` (контролы, слушатели, текстуры,
  рендерер) освобождается из единой цепочки `App.dispose()`; HMR в `main.ts`
  вызывает её при горячей перезагрузке.

## Конфигурация

Все настраиваемые параметры рендера — FOV (min/max/default), скорость вращения и
зума, демпинг, геометрия сферы, лимит DPR — собраны в `src/config/viewer.ts`.
В коде движка нет «магических чисел».
